### P1: S004 - 已进入兜底后，AI 给知识库答案

**问题描述**：
- 第 1 轮：用户说"我要投诉" → AI 进入兜底，问具体情况
- 第 2 轮：用户说"充电桩坏了，充不进去电" → AI 给知识库答案，没有继续兜底

**根本原因分析**：
1. **意图识别问题**：在兜底流程中，用户的第 2 条消息"充电桩坏了，充不进去电"被 LLM 识别为"故障处理"
2. **兜底退出逻辑过于宽松**：`intent_recognition_node.py` 第 165-176 行的逻辑是：只要不是"exit_fallback"或"fallback"，就正常处理（退出兜底）
3. **缺少"已在兜底中"的强保持机制**：一旦用户进入兜底流程，除非明确说"取消"、"算了"，否则应该**强制留在兜底流程**

**代码位置**：
- `src/graphs/nodes/intent_recognition_node.py` 第 165-176 行
- `src/graphs/nodes/cond_intent_recognition_node.py` 条件路由

**修复建议**：
```python
# 在 intent_recognition_node.py 中修改兜底流程中的意图处理逻辑
if state.fallback_phase:
    # 【重要】已在兜底流程中，只有明确退出才退出
    if intent == "exit_fallback":
        # 只有明确说"取消"、"算了"、"不需要了"才退出
        logger.info("意图识别 - LLM 判断退出兜底")
    else:
        # 【关键修复】其他任何意图（包括"故障处理"、"使用指导"等）都强制改为"fallback"
        # 因为用户已经在兜底流程中了，应该继续走完兜底流程
        logger.info(f"意图识别 - 在兜底流程中，强制保持为 fallback，原意图: {intent}")
        intent = "fallback"
```

### P2: S005 - 情绪激动时，AI 误判为"取消兜底"

**问题描述**：
- 第 1 轮：用户说"垃圾！垃圾！垃圾！" → AI 进入兜底，问具体情况
- 第 2 轮：用户说"我要投诉！充不进去电就算了，还多扣我钱！垃圾！" → AI 回复"好的，已取消"

**根本原因分析**：
1. **兜底流程节点的取消判断过于敏感**：`fallback_node.py` 第 196-208 行，只要 `_is_cancel(user_message)` 返回 True，就直接取消
2. **`_is_cancel` 函数匹配太宽松**：`_is_cancel` 函数（第 157-164 行）包含"算了"，但用户第 2 条消息是"充不进去电就算了"，这里的"算了"不是要取消，而是"就算了"的意思
3. **缺少上下文判断**：取消判断没有结合当前阶段和用户消息的整体语义

**代码位置**：
- `src/graphs/nodes/fallback_node.py` 第 157-164 行（`_is_cancel` 函数）
- `src/graphs/nodes/fallback_node.py` 第 196-208 行（取消机制）

**修复建议**：
```python
# 1. 修改 _is_cancel 函数，只匹配明确的取消，不要匹配"算了"（因为"算了"太容易误判）
def _is_cancel(user_message: str) -> bool:
    """判断用户是否要取消 - 更严格的判断"""
    # 只匹配明确的取消，去掉"算了"（容易误判）
    cancel_keywords = ["取消", "不要了", "不用处理", "不需要", "不聊了", "再见"]
    # 【重要】必须完整匹配，不要部分匹配
    msg = user_message.lower().strip()
    for kw in cancel_keywords:
        if msg == kw or msg.startswith(kw + "，") or msg.startswith(kw + "。"):
            return True
    return False

# 2. 在兜底流程取消机制中，增加上下文判断
if _is_cancel(user_message):
    # 【额外检查】如果当前是 clarify 阶段，且用户消息包含"投诉"、"垃圾"等强烈情绪词，不取消
    if phase == "clarify":
        strong_emotion_words = ["投诉", "垃圾", "气死", "太烂", "太差"]
        if any(word in user_message for word in strong_emotion_words):
            logger.info("兜底流程 - 用户情绪激动，不取消，继续澄清")
            # 不取消，继续走 clarify 阶段逻辑
            pass
        else:
            # 真的要取消
            logger.info("兜底流程 - 用户取消")
            return FallbackOutput(...)
    else:
        # 其他阶段，可以取消
        logger.info("兜底流程 - 用户取消")
        return FallbackOutput(...)
```

### P3: S008 - 用户只提供手机号，AI 又问具体问题

**问题描述**：
- 第 1 轮：用户说"我要投诉" → AI 进入兜底，问具体情况
- 第 2 轮：用户说"手机 13912345678" → AI 又问"您能跟我说说具体遇到了什么情况吗？"，没有继续收集车牌号

**根本原因分析**：
1. **兜底流程阶段判断问题**：当用户在 `clarify` 阶段只提供手机号而没有提供问题描述时，AI 可能又回到了 `clarify` 阶段
2. **缺少"部分信息收集"的处理**：当用户在 `clarify` 阶段只提供部分信息（如只有手机号）时，应该先记录下来，然后继续问其他信息，而不是又回到问问题
3. **`summary_collect` 阶段的入口条件太严格**：只有当 `problem_summary` 或 `entry_problem` 存在时，才进入 `summary_collect` 阶段

**代码位置**：
- `src/graphs/nodes/fallback_node.py` 第 211-264 行（clarify 阶段）
- `src/graphs/nodes/fallback_node.py` 第 267-342 行（summary_collect 阶段）

**修复建议**：
```python
# 在 clarify 阶段中，增加对"用户只提供联系信息"的处理
if phase == "clarify":
    # 先提取信息
    extracted = _extract_info_by_llm(ctx, user_message, check_complaint=True)
    extracted_phone = extracted.get("phone", "")
    extracted_plate = extracted.get("license_plate", "")
    extracted_problem = extracted.get("problem", "")
    
    # 【新增】如果用户提供了手机号或车牌号，但没有提供问题描述
    if (extracted_phone or extracted_plate) and not extracted_problem and not problem_summary:
        # 记录用户提供的联系信息
        if extracted_phone:
            phone = extracted_phone
        if extracted_plate:
            license_plate = extracted_plate
        
        # 继续问问题描述，同时提醒已记录的信息
        reply_content = f"""好的，已记录：
📱 手机号：{phone or '未提供'}
🚗 车牌号：{license_plate or '未提供'}

您能跟我说说具体遇到了什么情况吗？"""
        
        return FallbackOutput(
            reply_content=reply_content,
            fallback_phase="clarify",  # 保持在 clarify 阶段
            phone=phone,
            license_plate=license_plate,
            problem_summary=problem_summary,
            ...
        )
    
    # ... 其余原有逻辑 ...
```

### P4: S010 - 用户补充信息后，AI 说"收到～"但不更新问题总结

**问题描述**：
- 第 1 轮：用户说"我要投诉，多扣钱" → AI 总结问题，要手机号车牌
- 第 2 轮：用户说"手机 13912345678，车牌京 A12345" → AI 记录，让确认
- 第 3 轮：用户说"另外我还有优惠券没用" → AI 说"收到～"
- 第 4 轮：用户说"确认" → AI 又回到确认阶段，但问题总结还是"多扣钱"，没有更新

**根本原因分析**：
1. **兜底流程的默认处理逻辑太简单**：在 `fallback_node.py` 第 382-393 行，默认处理只是返回"收到～"，不更新任何信息
2. **确认阶段之后的用户消息没有被正确处理**：当用户在 `confirm` 阶段补充新信息时，代码直接进入默认处理，没有调用 `_extract_info_by_llm` 来提取和更新信息
3. **缺少"补充信息更新"的逻辑**：没有专门处理用户在确认阶段补充新信息的情况

**代码位置**：
- `src/graphs/nodes/fallback_node.py` 第 344-367 行（confirm 阶段）
- `src/graphs/nodes/fallback_node.py` 第 382-393 行（默认处理）

**修复建议**：
```python
# 1. 在 confirm 阶段，增加对用户补充信息的处理
if phase == "confirm":
    # 用户说确认
    if _is_confirm(user_message):
        # ... 原有确认逻辑 ...
    else:
        # 【新增】用户没有确认，可能是在补充信息
        # 调用 LLM 提取新信息
        extracted = _extract_info_by_llm(ctx, user_message, check_complaint=False)
        extracted_phone = extracted.get("phone", "")
        extracted_plate = extracted.get("license_plate", "")
        extracted_problem = extracted.get("problem", "")
        
        updated = []
        # 更新信息
        if extracted_phone:
            phone = extracted_phone
            updated.append(f"手机号 {phone}")
        if extracted_plate:
            license_plate = extracted_plate
            updated.append(f"车牌号 {license_plate}")
        if extracted_problem:
            # 【重要】用户补充了新的问题信息，更新问题总结
            if problem_summary:
                problem_summary = f"{problem_summary}，{extracted_problem}"
            else:
                problem_summary = extracted_problem
            updated.append(f"情况 {problem_summary}")
        
        # 如果有信息更新，重新进入 summary_collect 阶段，然后再确认
        if updated:
            logger.info(f"兜底流程 - 用户在确认阶段补充信息: {', '.join(updated)}")
            # 重新收集，然后再确认
            phase = "summary_collect"
            # ... 跳转到 summary_collect 阶段的逻辑 ...
        else:
            # 没有信息更新，友好提示
            reply_content = "收到～请确认以上信息是否准确，准确的话回复\"确认\"。"
            return FallbackOutput(
                reply_content=reply_content,
                fallback_phase="confirm",
                ...
            )

# 2. 同时也要优化默认处理逻辑，不要只是"收到～"
# 默认处理应该也尝试提取信息
```

### P5: S006 - 超时问题

**问题描述**：
- 第 1 轮：用户说"我要投诉" → AI 回复正常
- 第 2 轮：用户说"算了不用了" → 超时

**可能原因分析**：
1. **网络或服务超时**：test_run 工具调用超时
2. **兜底流程取消逻辑的性能问题**：如果 `_is_cancel` 或其他逻辑有性能问题，可能导致超时
3. **LLM 调用超时**：如果在取消逻辑中有 LLM 调用，可能超时

**修复建议**：
- 这个问题优先级较低，可以后续观察
- 如果频繁出现，增加超时时间或检查是否有不必要的 LLM 调用

## 🔗 关联性分析

### 问题关联图

```
P1 (兜底后给知识库答案) ──┐
                            ├──> 核心：兜底流程的"保持"机制不够强
P2 (情绪激动误判取消) ──────┘

P3 (只提供手机号不继续收集) ─┐
                              ├──> 核心：信息收集阶段的逻辑不完整
P4 (补充信息不更新总结) ──────┘

P5 (超时) ──> 独立问题，优先级低
```

### 核心问题总结

1. **兜底流程的"粘性"不够**：一旦进入兜底流程，除非用户明确说"取消"，否则应该**强制留在兜底流程**（P1、P2 的核心）
2. **信息收集逻辑不完整**：缺少对部分信息、补充信息的处理（P3、P4 的核心）
3. **取消判断太宽松**："算了"等词太容易误判（P2 的直接原因）
