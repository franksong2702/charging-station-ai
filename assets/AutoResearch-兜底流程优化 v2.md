# Auto-Research 兜底流程优化

> **版本**: v2.0 | **创建**: 2026-04-09  
> **核心理念**: AI 测试 AI + 自动迭代优化

## 一、测试框架（已验证✅）

```
┌─────────────────────────────────────────────────────────────┐
│              测试员 AI (Coze 编程助手)                        │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  1. 发送用户消息 → 被测 AI (兜底流程工作流)                   │
│  2. 获取 AI 回复 → 判断当前阶段                               │
│  3. 根据阶段 → 生成下一条用户消息                             │
│  4. 重复直到：工单创建 OR 测试失败                            │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**阶段判断规则**：

| AI 回复特征 | 判断为 | 下一条用户消息 |
|------------|--------|---------------|
| "什么问题"+"怎么回事"+"具体" | 询问阶段 | 描述具体问题 |
| "手机号"+"车牌号"+"提供" | 收集信息 | 提供手机号 + 车牌 |
| "确认吗"+"准确吗"+"以上信息" | 确认阶段 | "确认" 或 纠正 |
| "1-3 个工作日"+"尽快处理" | 已完成 | 测试通过 |
| 其他异常回复 | 异常 | 记录失败，停止 |

## 二、Auto-Research 迭代循环

```
┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│ 修改 Prompt   │ → │ 跑 10 个场景   │ → │ 计算通过率   │
│ 变体         │   │ 测试         │   │              │
└──────────────┘   └──────────────┘   └──────────────┘
       ↑                                      │
       │                                      ↓
       └────────── 保留高通过率版本 ────────────┘
```

### 可优化的 Prompt 变量

| 变量 | 可改动内容 | 影响 |
|------|-----------|------|
| **兜底触发词** | 添加/删除关键词（投诉、退款、垃圾等） | 触发准确率 |
| **意图识别** | 修改提示词措辞、示例 | 意图分类准确率 |
| **话术风格** | 亲切/正式/简洁 | 用户体验 |
| **确认关键词** | 扩展/精简确认词列表 | 确认环节通过率 |

## 三、10 个测试场景

### 场景 1-3：基础投诉（已验证）
1. 优惠券投诉："你们这个垃圾系统，气死我了"
2. 充电故障："什么垃圾服务，充不进去电"
3. 退款投诉："我要退款"

### 场景 4-6：边界情况
4. 模糊投诉："我要投诉"（不说具体问题）
5. 情绪激动："垃圾！垃圾！垃圾！"（重复 3 次）
6. 中途取消："我要投诉...算了不用了"

### 场景 7-10：复杂场景
7. 用户纠正："不对，你总结得不对，我实际是..."
8. 分次提供："手机 13912345678"→"车牌京 A12345"（分两轮）
9. 语音格式："手机号 139。16425678。车牌。沪 A Dr 3509"
10. 补充信息："另外我还有优惠券没用"

## 四、执行指令

### 第一轮：基准测试

```
你好，请执行兜底流程基准测试。

【任务】
1. 用 AI 测试 AI 方法，跑 10 个测试场景
2. 每个场景完整走完流程（直到工单创建或失败）
3. 记录每轮对话和判断阶段
4. 生成测试报告

【输出格式】
## 测试结果

| 场景 | 结果 | 轮数 | 工单创建 |
|------|------|------|----------|
| 场景 1 | ✅ | 4 轮 | 是 |
| 场景 2 | ❌ | 2 轮 | 否 |
...

通过率：X/10 = XX%

## 失败分析
（每个失败场景的详细分析）
```

### 第二轮：Prompt 优化后测试

```
你好，请修改意图识别 Prompt（config/intent_recognition_llm_cfg.json）。

【修改建议】
1. 扩展兜底触发词（添加"垃圾"、"退款"、"赔偿"等）
2. 优化阶段判断提示词
3. 添加边界情况示例

修改后，重新跑 10 个测试场景，对比通过率。
```

### 第三轮：Auto-Research 循环

```
你好，请执行 Auto-Research 自动迭代。

【循环逻辑】
1. 修改 Prompt（改动一个变量）
2. 跑 10 个场景测试
3. 计算通过率
4. 如果通过率提升 → 保留版本
5. 如果通过率下降 → 回滚
6. 重复 5 轮

【输出】
- 每轮 Prompt 修改内容
- 每轮通过率对比
- 最终推荐版本
```

## 五、评分标准

| 维度 | 权重 | 计算方式 |
|------|------|---------|
| **场景通过率** | 50% | 通过场景数 / 总场景数 |
| **平均轮数** | 20% | 越接近 4 轮越好（标准流程） |
| **工单完整度** | 30% | 手机号、车牌号、问题总结齐全 |

**综合得分** = 场景通过率×0.5 + (4/平均轮数)×0.2 + 工单完整度×0.3
