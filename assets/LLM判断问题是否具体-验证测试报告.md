# LLM 判断问题是否具体 - 验证测试报告

> 测试时间：2026-04-09
> 优化内容：用 LLM 替代硬编码规则，判断问题是否足够具体

---

## 🎯 优化目标

### 问题分析
之前用硬编码规则（判断长度、判断关键词）来判断问题是否足够具体，很奇怪且不灵活。

### 解决方案
修改了 `fallback_node.py`，用大语言模型来辅助决策，判断问题是否足够具体：

1. **用 LLM 判断的逻辑**：
   - 提示词让 LLM 判断用户的问题描述是否足够具体
   - 如果问题描述比较模糊（只有"充不进去电"、"多扣钱"等简单描述）→ 需要更多细节
   - 如果问题描述包含时间、地点、具体情况等信息 → 不需要更多细节
   - 返回 JSON 格式：`{"need_more_details": true/false}`

2. **降级处理**：
   - 如果 LLM 调用失败，默认需要更多细节
   - 确保流程不会因为 LLM 失败而中断

---

## 🧪 验证测试结果

### 场景 1：用户只说"我要投诉充不进去电"（信息不够）
**用户**："我要投诉充不进去电"
**AI**："哦，明白了！您说的是：充不进去电\n\n为了更好地帮您处理，能再跟我说一说具体情况吗？比如：\n- 什么时候发生的？\n- 在哪个站点？\n- 具体是什么情况？"
**结果**：✅ LLM 判断需要更多细节，AI 进一步追问

### 场景 2：用户提供更多细节（信息足够）
**用户**："今天下午3点多在奥林匹克公园站，充到80%的时候突然跳枪了，然后就再也充不进去了"
**AI**："哦，那我明白了，您的问题大概是这样的：充不进去电，今天下午3点多在奥林匹克公园站充电，充到80%的时候突然跳枪，之后再也充不进去了\n\n您的问题我们会反馈给专业的客服团队去处理。请您留下手机号和车牌号，方便我们的客服后续联系您。"
**结果**：✅ LLM 判断不需要更多细节，AI 更新问题总结，进入收集信息阶段

### 场景 3：用户只说"我要投诉多扣钱"（信息不够）
**用户**："我要投诉多扣钱"
**AI**："哦，明白了！您说的是：多扣钱\n\n为了更好地帮您处理，能再跟我说一说具体情况吗？比如：\n- 什么时候发生的？\n- 在哪个站点？\n- 具体是什么情况？"
**结果**：✅ LLM 判断需要更多细节，AI 进一步追问

### 场景 4：用户提供更多细节（信息足够）
**用户**："昨天晚上在望京西站，充了20分钟实际扣了1小时的钱，而且我的优惠券也没用上"
**AI**："哦，那我明白了，您的问题大概是这样的：多扣钱，在望京西站充电20分钟实际被扣1小时的费用，优惠券未用上\n\n您的问题我们会反馈给专业的客服团队去处理。请您留下手机号和车牌号，方便我们的客服后续联系您。"
**结果**：✅ LLM 判断不需要更多细节，AI 更新问题总结，进入收集信息阶段
