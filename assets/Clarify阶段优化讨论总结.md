# Clarify 阶段优化讨论总结

> 讨论时间：2026-04-09
> 参与讨论：用户、工作流搭建专家

---

## 📋 问题背景

### 发现的问题
用户观察到当前工作流存在"无条件兜底"问题：
1. 用户一投诉/一要求退款，AI 就直接兜底
2. 问题总结太简略（如"用户要求退款"）
3. 直接问手机号和车牌号，一句多余的话都没有
4. 没有先询问更详细的情况，没有试图帮用户解决问题

### 理想的客服流程
1. **先询问详细情况**：用户有诉说欲，想把事情说清楚
2. **多次对话收集信息**：和用户多聊几句，收集更多信息
3. **试图帮用户解决**：在可能的情况下，先帮用户解决问题
4. **最终总结问题**：信息收集完整后，再总结问题
5. **用户确认**：总结后让用户确认
6. **胡搅蛮缠型用户处理**：如果用户死活不说清楚，可以直接拒掉

---

## 💡 优化方向讨论

### 核心优化点
1. **根据用户回答灵活判断**：不是固定几轮
2. **总结要详细**：要包含原因（如"因为优惠券没使用，所以多扣了钱要求退款"）
3. **Clarify 阶段要有确认步骤**：用户确认了总结后，再进入下一阶段
4. **参考知识库给有效回答**：如果可以的话，参考知识库给用户有效回答

### 技术难点
**用户指出的难点**：
- 总结要详细、包含原因，可能需要大模型介入来判断
- 这一块的实现比较 tricky

---

## 🤔 讨论过的方案

### 方案 A：优化现有 LLM 提取（最小改动）
**改动最小**：
- 优化 `_extract_info_by_llm` 函数
- 让 LLM 同时生成"详细总结"和"简单问题"
- 在 clarify 阶段增加确认逻辑

**优点**：
- 改动最小，不需要新增节点
- 利用现有的 LLM 调用

**缺点**：
- 单个 LLM 调用要做的事情太多（提取信息 + 生成总结）
- 可能不够灵活

---

### 方案 B：新增 Summary Agent 节点（中等改动）
**新增一个节点**：
- 在 clarify 阶段之后，新增一个 `summary_agent` 节点
- 专门用 LLM 生成详细总结
- 专门处理确认逻辑
- 可以参考知识库

**优点**：
- 职责更清晰
- 更灵活
- 可以单独优化这个节点

**缺点**：
- 需要新增节点
- 改动比方案 A 大

**当前倾向**：工作流搭建专家倾向于方案 B

---

### 方案 C：完全重新设计 Clarify 阶段（大改动）
**完全重新设计**：
- clarify 阶段完全用 LLM 驱动
- LLM 决定问什么问题
- LLM 生成总结
- LLM 判断是否需要确认

**优点**：
- 最灵活
- 可以处理最复杂的场景

**缺点**：
- 改动最大
- 风险最高
- 需要更多提示词工程
