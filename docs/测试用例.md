# 充电桩智能客服测试用例

## 版本信息
- 版本：v2.0
- 更新日期：2024年
- 更新内容：新增兜底流程、工单系统、优化意图识别

---

## 测试环境准备

### 必需参数
| 参数 | 说明 | 示例 |
|------|------|------|
| user_id | 用户身份标识（用于多轮对话） | test_user_001 |
| user_message | 用户发送的消息 | "充电桩怎么用？" |
| conversation_history | 对话历史（JSON数组） | [{"role": "user", "content": "..."}] |
| fallback_phase | 兜底流程阶段（可选） | collect_info / confirm |

---

## 测试用例列表

### 一、正常问答场景

#### TC-001：使用指导
**目的**：测试用户询问充电桩使用方法

**输入**：
```json
{
  "user_message": "充电桩怎么用？",
  "user_id": "test_user_001"
}
```

**预期输出**：
- 回复包含充电桩使用方法的指导
- 不触发评价请求
- 回复示例："您好，使用充电桩的完整流程很简单..."

---

#### TC-002：故障处理
**目的**：测试用户报告故障问题

**输入**：
```json
{
  "user_message": "充电枪拔不出来怎么办？",
  "user_id": "test_user_001"
}
```

**预期输出**：
- 回复包含故障处理建议
- 不触发评价请求

---

#### TC-003：投诉咨询
**目的**：测试用户咨询投诉相关问题

**输入**：
```json
{
  "user_message": "刚才充电扣费不对，想退款",
  "user_id": "test_user_001"
}
```

**预期输出**：
- 回复引导用户提供详细信息
- 可能触发兜底流程

---

### 二、轻度不满场景

#### TC-010：轻度不满
**目的**：测试用户表达轻度不满时，AI继续尝试帮助

**输入**：
```json
{
  "user_message": "这回答没用",
  "user_id": "test_user_001",
  "conversation_history": [
    {"role": "user", "content": "充电桩怎么用？"},
    {"role": "assistant", "content": "您好，使用充电桩的完整流程很简单..."}
  ]
}
```

**预期输出**：
- 回复道歉并请求详细描述
- 回复示例："很抱歉没能帮到您 😔 您能再详细描述一下遇到的问题吗？我再尝试帮您解决。"
- 记录不满意反馈到数据库

---

#### TC-011：轻度不满后补充问题
**目的**：测试用户补充问题后，AI继续帮助

**输入**：
```json
{
  "user_message": "我找不到二维码在哪里",
  "user_id": "test_user_001",
  "conversation_history": [
    {"role": "user", "content": "充电桩怎么用？"},
    {"role": "assistant", "content": "您好..."},
    {"role": "user", "content": "这回答没用"},
    {"role": "assistant", "content": "很抱歉...请详细描述问题"}
  ]
}
```

**预期输出**：
- AI针对具体问题"找不到二维码"给出解答
- 不触发兜底流程

---

### 三、兜底流程场景

#### TC-020：强烈不满触发兜底
**目的**：测试用户表达强烈不满时，触发兜底流程

**输入**：
```json
{
  "user_message": "太差了，我要转人工",
  "user_id": "test_user_001",
  "conversation_history": [
    {"role": "user", "content": "充电桩怎么用？"},
    {"role": "assistant", "content": "您好..."}
  ]
}
```

**预期输出**：
- 回复提示会记录问题并反馈给工作人员
- 请求用户提供手机号和车牌号
- 回复示例："好的，我们会将您的问题记录下来，并反馈给相关工作人员处理..."

---

#### TC-021：用户提供联系信息
**目的**：测试兜底流程中用户提供联系信息

**输入**：
```json
{
  "user_message": "手机13812345678，车牌京A12345",
  "user_id": "test_user_001",
  "fallback_phase": "collect_info",
  "conversation_history": [
    {"role": "user", "content": "太差了，我要转人工"},
    {"role": "assistant", "content": "好的...请提供手机号和车牌号"}
  ]
}
```

**预期输出**：
- 提取手机号：13812345678
- 提取车牌号：京A12345
- 生成问题总结
- 请求用户确认

---

#### TC-022：用户确认问题总结
**目的**：测试用户确认问题总结后创建工单

**输入**：
```json
{
  "user_message": "确认",
  "user_id": "test_user_001",
  "fallback_phase": "confirm",
  "phone": "13812345678",
  "license_plate": "京A12345",
  "problem_summary": "用户遇到充电桩使用问题...",
  "conversation_history": [...]
}
```

**预期输出**：
- 创建工单
- 发送邮件到客服邮箱
- 回复："✅ 您的问题已成功提交！工单编号：X"

---

#### TC-023：用户补充问题

```json
  "user_message": "我是在朝阳区的充电桩遇到的问题",

  "license_plate": "京A12345",
```

**预期输出**：
- 更新问题总结，包含用户补充的内容
- 再次请求用户确认

---

#### TC-024：用户要求转人工

**输入**：

## 数据库验证

### dialog_records 表
验证以下场景的记录是否正确保存：
- 评价反馈（满意/不满意）
- 不满意反馈

### case_records 表
验证兜底流程创建的工单是否包含：
- user_id
- phone
- license_plate
- problem_summary
- conversation_context
- status = 'pending'

---

## 注意事项

1. **测试顺序**：建议按测试用例编号顺序执行，确保数据依赖正确
2. **数据清理**：测试前可清理测试用户的历史数据
3. **邮件检查**：兜底流程测试后，检查邮箱是否收到邮件
4. **日志查看**：如有问题，查看 /app/work/logs/bypass/app.log 日志
